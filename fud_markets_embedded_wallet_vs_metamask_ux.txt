FUD.markets — UX recomendada para embedded wallet de Privy + MetaMask detectada
===============================================================================

Contexto
--------

Tengo un prediction market (FUD.markets) con Privy como auth. Cuando un user se registra con Google/Email, Privy crea una embedded wallet automáticamente. Esa wallet es la primary wallet del user.

PROBLEMA: el user también tiene MetaMask instalado con otra wallet.
Mi app detecta ambas:
- Privy embedded wallet: 0x99e7... (primary, guardada en DB)
- MetaMask wallet: 0x621E... (browser connected)

Esto genera un banner "Connected wallet doesn't match your linked wallet" que confunde al user nuevo.


Respuesta 1
-----------

Sí: para un user nuevo que entra con Google/Email, tu UX debería tratar a la embedded wallet de Privy como la wallet activa de trading y no mostrar un error por tener MetaMask instalada con otra address. Privy distingue entre wallets linked y connected; un usuario puede tener embedded y external al mismo tiempo, y eso no implica que haya un problema. Además, las embedded wallets de Privy pueden firmar EIP-712, y Privy las presenta como self-custodial con opción de exportar la key.

Recomendación concreta
----------------------

### 1) Qué hacer con MetaMask en el caso normal
Ignorala para el flujo principal.
Si el user se registró con Google/Email y tu backend ya guardó la embedded como `primary_wallet_address`, entonces:

- esa embedded wallet es la trading wallet
- esa embedded wallet es la que firma bets
- esa embedded wallet es la que recibe depósitos para luego fondear el vault
- MetaMask puede existir, pero no debe “pisar” la wallet activa por default

No muestres el banner “Connected wallet doesn't match your linked wallet” en onboarding. Ese banner comunica “hay un error”, cuando en realidad hay dos wallets distintas con roles distintos. Privy justamente documenta que puede haber wallets embedded y external simultáneamente.

En vez de ese banner, mostraría algo como:

Trading wallet: 0x99e7…
MetaMask detected: 0x621E…
Want to use MetaMask instead? Switch wallet

Eso baja muchísimo la confusión.

### 2) Si quiere usar MetaMask en vez de la embedded
Sí, hacelo como un flujo explícito con un botón tipo:

Use MetaMask as trading wallet instead

Pero no como acción silenciosa. Tiene que ser una decisión consciente, porque tu vault y tus balances están atados a una address.

Regla de producto:
No cambies la `primary_wallet_address` automáticamente solo porque MetaMask está instalada o conectada.

Flujo correcto para “switch”:
1. El user toca “Use MetaMask instead”
2. Le mostrás una explicación:
   - “Your trading balance is tied to your current wallet address.”
   - “If you switch, future deposits and trades will use your MetaMask wallet.”
3. Si la embedded tiene:
   - vault balance > 0, o
   - posiciones abiertas o fondos que dependen de esa address,
   entonces no permitas el switch instantáneo
4. Pedile primero:
   - withdraw del vault a la embedded
   - cerrar lo que corresponda
   - mover fondos a la nueva wallet si hace falta
5. Recién después:
   - cambiás `primary_wallet_address`
   - marcás MetaMask como nueva trading wallet

Regla simple para v1:
- si no tiene fondos ni actividad: permitir switch inmediato
- si ya tiene fondos/actividad: bloquear switch y mostrar pasos

### 3) Cómo fondear la embedded
Tu mejor UX para un user nuevo con Google + MetaMask instalada es:

CTA principal:
Add funds to your trading wallet

Y adentro tres caminos:

Transfer USDC
Mostrás la embedded address, Base, USDC, y botones de copiar.

Buy USDC
Si ya tenés onramp bien integrado.

Send from MetaMask
Sí, esto lo facilitaría. No tenés que depender de que el user “entienda” que debe copiar una address y cambiar de app. Podés mostrar:

- Trading wallet: `0x99e7...`
- Network: Base
- Token: USDC
- Botón: Copy address
- Botón: Open MetaMask
- Texto corto: “Send USDC on Base from your MetaMask wallet to your trading wallet”

Eso resuelve exactamente el caso que describís.

### 4) Quién debería firmar los bets EIP-712
Para ese user nuevo, la firma debería hacerla la embedded wallet.

### Por qué es mejor UX
- no depende de qué wallet de MetaMask esté seleccionada
- no requiere popup externo
- no rompe en mobile o en browsers sin extensión
- mantiene consistente la address que firma con la address que tiene el balance en vault

En tu modelo, la regla correcta debería ser:

la wallet que firma bets = la trading wallet = la primary wallet

Para un signup con Google/Email, eso significa embedded wallet por default.

MetaMask solo debería firmar bets si el usuario hizo el flujo explícito de cambiar su trading wallet a la external.

### 5) ¿La embedded persiste si desinstala MetaMask o cambia de browser?
Sí, en general sí.

O sea:
- si desinstala MetaMask, no pierde la embedded
- si cambia de browser, puede seguir entrando con Google/Email
- si cambia de device, Privy puede reprovisionar la wallet al autenticarse
- si querés más seguridad, activás recovery/MFA para nuevos dispositivos

Además, Privy permite exportar la private key de la embedded y usar esa misma address en MetaMask si quiere.

Eso te da una salida elegante:
- onboarding fácil con embedded
- soberanía avanzada después, si el user la quiere

Respuestas directas
-------------------

### 1. ¿Ignorar MetaMask cuando el user tiene embedded?
Sí, para el flujo principal de un user nuevo.
No la ignores técnicamente, pero sí ignorala como wallet activa de trading salvo que el user elija cambiar.

### 2. ¿Cómo switchear la primary wallet?
Con un botón explícito: “Use external wallet instead”.
Y solo permitir cambio directo si no hay fondos/actividad. Si ya hay balance/posición ligada a la embedded, primero withdraw / vaciar / cerrar lo necesario.

### 3. ¿Cómo fondear la embedded?
Las tres buenas opciones son:
- transferir USDC desde MetaMask a la embedded
- usar onramp de Privy si ya está listo
- facilitar el envío mostrando address, red y botones para copiar/abrir MetaMask

### 4. ¿Quién firma los bets EIP-712?
La embedded wallet, para el user que arrancó con Google/Email.

### 5. ¿La embedded persiste sin MetaMask y en otro device?
Sí, normalmente sí.

Recomendación final en una línea
--------------------------------

Para un user nuevo que entra con Google y tiene MetaMask instalada: tratá la embedded wallet como única trading wallet, eliminá el banner de mismatch, facilitá “send from MetaMask” para fondearla, y ofrecé “switch to MetaMask” solo como acción avanzada y explícita.

Lógica que elimina el banner confuso
------------------------------------

## Hoy probablemente tenés algo así
Si:
- wallet del browser conectada != wallet guardada en DB
→ mostrar error

## Cambialo por esto

### Regla nueva
Mostrar warning solo si:
- `trading_wallet_type = external`
- y la wallet externa activa del browser no coincide con la `trading_wallet_address`

Ejemplo:
- user eligió MetaMask como trading wallet
- pero MetaMask está parada en otra address
- ahí sí corresponde warning

### No mostrar warning si:
- `trading_wallet_type = embedded`
- y además MetaMask está instalada o conectada con otra address

Porque en ese caso MetaMask no es la wallet activa.

Pseudológica simple
-------------------

```ts
const tradingWallet = user.tradingWalletAddress
const tradingWalletType = user.tradingWalletType // embedded | external
const metamaskAddress = browserWallets.metamask?.address

const shouldShowExternalMismatch =
  tradingWalletType === "external" &&
  !!metamaskAddress &&
  metamaskAddress.toLowerCase() !== tradingWallet.toLowerCase()

const shouldShowMetamaskDetectedInfo =
  tradingWalletType === "embedded" &&
  !!metamaskAddress &&
  metamaskAddress.toLowerCase() !== tradingWallet.toLowerCase()
```

### `shouldShowMetamaskDetectedInfo = true`
Mostrar info neutra:

Your trading wallet is still your embedded wallet.
