FUD.markets — Research y propuesta de escrow contract para prediction market
==========================================================================

Objetivo
--------
Armar un smart contract escrow para un prediction market crypto tipo parimutuel LONG/SHORT,
priorizando código auditado o battle-tested en producción, y bajar eso a una propuesta
práctica para shipping rápido en MVP.

Resumen ejecutivo
-----------------
Conclusión principal: no apareció un contrato público que matchee 1:1 la spec pedida
(single master contract, USDC ERC-20 only, parimutuel payouts, resolver permissioned,
cancel/refund, fee a treasury, no upgradeability) y que además sea auditado y claramente
battle-tested con volumen real.

Sí aparecieron muy buenas bases para adaptar rápido:

1. Azuro V2
   - Mejor referencia de arquitectura tipo singleton/pool pattern
   - Auditada y usada de verdad
   - Más parecida a la arquitectura buscada

2. Gnosis Conditional Tokens
   - Base muy battle-tested para collateral + resolution
   - Muy usada indirectamente vía Polymarket
   - No sirve directo para payout parimutuel

3. Polymarket (CTF Exchange + UMA adapter)
   - Súper battle-tested
   - Muy fuerte en settlement/oracle separation
   - Pero es exchange / CLOB + CTF, no escrow parimutuel simple

Recomendación final:
No forkear un protocolo entero. Hacer un contrato nuevo, chico, no-upgradeable,
inspirado principalmente en:
- Azuro para market lifecycle / singleton pool / roles
- Polymarket UMA adapter para resolver/oracle separation
- OpenZeppelin para SafeERC20, ReentrancyGuard, permisos y seguridad operacional

-----------

Auditoría:

- Mejor base conceptual para extraer patrones

==========================================================================
PROPUESTA EXACTA DE CONTRATO PARA FUD V1
==========================================================================

Nombre sugerido:
- ParimutuelEscrow.sol

Características:
- Un solo contrato maestro
- ERC-20 only (USDC / USDC.e)
- Sin ETH nativo
- Sin upgradability
- Resolver permissioned
- Fee configurable (default 5%)
- Treasury configurable
- Cancel / refund manual
- Claim pull-based
- Sin leverage
- Sin liquidations
- Sin funding rates
- Sin order book
- Sin AMM
- Sin ERC1155 positions

------------------------------------------------------------
Enums
------------------------------------------------------------

enum Side { NONE, LONG, SHORT }
enum Status { NONE, OPEN, RESOLVED, CANCELED }

------------------------------------------------------------
Storage principal
------------------------------------------------------------

struct Market {
    uint64 closesAt;
    Status status;
    Side winner;
    uint128 longPool;
    uint128 shortPool;
    uint128 feeAmount;
    uint128 payoutPool;
    int256 exitPrice;
}

struct Position {
    uint128 longAmount;
    uint128 shortAmount;
    bool claimed;
}

mapping(bytes32 => Market) public markets;
mapping(bytes32 => mapping(address => Position)) public positions;

IERC20 public immutable collateral;
address public treasury;
address public resolver;
uint16 public feeBps;
uint256 public accruedTreasuryFees;

------------------------------------------------------------
Flujo funcional
------------------------------------------------------------

1) createMarket(bytes32 marketId, uint64 closesAt)
- onlyOwner o onlyBackend
- valida que el market no exista
- valida closesAt > block.timestamp
- registra market OPEN

2) bet(bytes32 marketId, Side side, uint128 amount)
- market OPEN
- block.timestamp < closesAt
- amount > 0
- side = LONG o SHORT
- user hizo approve() antes
- safeTransferFrom(msg.sender, address(this), amount)
- incrementa pool correcto
- incrementa position del usuario

3) resolve(bytes32 marketId, Side winner, int256 exitPrice)
- onlyResolver
- block.timestamp >= closesAt
- status == OPEN
- calcula:
  - winnerPool
  - loserPool
  - feeAmount = loserPool * feeBps / 10_000
  - payoutPool = loserPool - feeAmount
- suma feeAmount a accruedTreasuryFees
- fija winner, exitPrice y status = RESOLVED

4) cancel(bytes32 marketId)
- onlyResolver
- status == OPEN
- status = CANCELED

5) claim(bytes32 marketId)
- nonReentrant
- !claimed
- si market CANCELED:
  payout = longAmount + shortAmount
- si market RESOLVED:
  toma solo el side ganador
  payout = userStake + userStake * payoutPool / winnerPool
- marca claimed = true
- safeTransfer(msg.sender, payout)

6) sweepFees(address to, uint256 amount)
- onlyOwner
- transfiere fees acumulados a treasury o destino autorizado

==========================================================================
MECÁNICA DE PAYOUT
==========================================================================

Tu fórmula deseada:

amount + (loserPool * 0.95 / winnerPool) * amount

Implementación equivalente más limpia:

loserPool = (winner == LONG) ? shortPool : longPool
winnerPool = (winner == LONG) ? longPool : shortPool

feeAmount = loserPool * feeBps / 10_000
payoutPool = loserPool - feeAmount

bonus = userStake * payoutPool / winnerPool
payout = userStake + bonus

Ventaja:
- feeAmount y payoutPool quedan congelados al resolver
- si se cambia feeBps después, no afecta mercados viejos
- eso simplifica auditoría y evita inconsistencias

==========================================================================
EDGE CASES IMPORTANTES
==========================================================================

1) Winner pool vacío
--------------------
Si resolve() marca LONG pero longPool == 0, no hay ganadores.

Recomendación:
- revertir resolve() si winnerPool == 0
- usar cancel() en ese caso

2) Resolver malicioso / bug de backend
--------------------------------------
Para v1:
- owner puede cambiar resolver
- owner puede pausear
- resolver puede cancel()
- MarketResolved debe emitir exitPrice y datos suficientes para auditoría offchain

3) ERC-20 no estándar
---------------------
Usar SafeERC20 siempre.

4) Dust por rounding
--------------------
Va a existir por división entera.

Recomendación simple para v1:
- dejar dust residual en el contrato
- opcionalmente barrerlo más adelante con lógica muy conservadora
- no complicar demasiado el MVP por esto

5) Doble claim
--------------
Cada posición debe poder reclamar una sola vez.
Flag claimed obligatorio.

==========================================================================
PERMISOS Y ROLES
==========================================================================

Roles mínimos:
- owner
- resolver

Owner:
- createMarket
- setResolver
- setTreasury
- setFeeBps
- pause
- unpause
- sweepFees

Resolver:
- resolve
- cancel

Por qué:
- Tu MVP hoy resuelve de forma permissioned
- La separación owner / resolver deja abierta la puerta para después enchufar
  un adapter con backend signer, UMA, Pyth, etc.

==========================================================================
INVARIANTS / TESTS QUE SÍ O SÍ HAY QUE HACER
==========================================================================

1. No se puede apostar después de closesAt
2. No se puede resolver antes de closesAt
3. No se puede resolver dos veces
4. No se puede cancelar dos veces
5. No se puede resolver un market cancelado
6. No se puede cancelar un market resuelto
7. Cada user solo puede claim una vez
8. En cancel, cada user recupera exactamente su stake
9. En resolve, el total pagado a ganadores + treasury fee nunca supera los fondos del market
10. Si winnerPool == 0, resolve debe fallar o el market debe cancelarse
11. feeBps no puede ser absurdo (ej. cap razonable)
12. safeTransfer / safeTransferFrom deben funcionar con USDC real
13. Reentrancy en claim no debe ser posible

la respuesta honesta es:

No hay un template público perfecto que matchee exactamente FUD.markets v1.
Lo más inteligente es:
- escribir un contrato nuevo chico
- inspirado en patrones de Azuro, Polymarket y OpenZeppelin
- con surface area mínima
- sin upgradeability
- con payout parimutuel fijo
- con roles claros
- con accounting congelado al resolver
- con tests serios en Foundry

Ese camino maximiza:
- velocidad de shipping
- claridad para auditar
- cercanía real con tu producto
- menor riesgo de meter complejidad innecesaria
