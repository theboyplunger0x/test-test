RFC técnico

# RFC — FUD.markets Account-Based Balance Architecture

## Status
Draft

## Owner
FUD.markets

## Purpose
Define the target architecture for FUD balances so that balances belong to the user account, not to whichever wallet the user connected most recently.

## 1. Summary
FUD currently ties balances to `walletAddress` inside `FUDVault.sol`.

That creates a product-breaking issue:
- if the user changes wallet, the balance appears to change
- if the user connects another wallet, they see another balance
- the balance behaves as if it belongs to the wallet session, not the user account

The new architecture keeps the vault balance onchain by address, but changes which address represents the account.

### Core decision
Each FUD account will have a stable Main Wallet.

That wallet will:
- be created/recovered through Privy
- remain the canonical wallet for the account
- own the onchain balance in `FUDVault`
- sign bets and withdrawals
- remain stable regardless of login method

All other wallets will be treated as linked wallets only.

They may be used for:
- login
- funding
- withdrawal destination
- recovery/future interoperability

They will not define the account balance.

## 2. Problem Statement
### Current state
- `FUDVault.sol` stores `balances[address]`
- login method may be Google / Email / external wallet
- in some flows, the active wallet address becomes the effective balance owner
- external browser wallet state leaks into core product logic

### Product consequences
- balance appears wallet-dependent
- users can lose access to “their” balance when wallet context changes
- connecting another wallet creates confusion
- “wrong wallet connected” states become common
- account identity and wallet identity are mixed

### Root cause
The issue is not Privy.

The issue is that the system currently treats:
- wallet = account
instead of:
- account = account
- wallet = linked tool

## 3. Goals
### Primary goals
- balance must be independent from the wallet currently connected in browser
- user can fund from any wallet
- changing linked wallet must not change visible balance
- user should not need gas for core product flows
- architecture should remain crypto-native
- architecture should remain compatible with Privy
- should avoid a fully custodial DB-ledger model as source of truth

### Secondary goals
- preserve as much of the current onchain model as possible
- stay directionally close to Polymarket
- support future linked-wallet UX
- keep room for operator/relayer-based flows

### Non-goals
- fully redesigning settlement from scratch
- introducing a pure custodial exchange model
- replacing Privy
- rebuilding the system around `userIdHash` balances immediately

## 4. Architectural Decision
### 4.1 Chosen model
One FUD account = one stable Main Wallet = one canonical onchain balance owner

The vault may continue to store:

```solidity
mapping(address => uint256) public balances;
```

But that address must always be:
- `main_wallet_address`

not:
- the browser wallet currently connected
- the last wallet used to log in
- the wallet the user happened to choose in MetaMask today

### Result
At the product level, balance becomes “account-based”.

At the contract level, balance remains “address-based”.

The key is that the address is now a stable account wallet, not a variable session wallet.

## 5. Why this model
This model gives FUD the best tradeoff across:
- account consistency
- onchain auditability
- low product complexity
- compatibility with Privy
- future scaling
- resemblance to Polymarket’s proxy-wallet model

### Why not pure DB balance
A DB balance as source of truth would make the backend custodial de facto.

That would mean:
- backend decides real balance
- backend decides withdrawals
- onchain state becomes secondary
- user cannot independently verify the balance model as easily

This may be practical for some exchanges, but it is a worse fit for a crypto-native prediction market if it can be avoided.

### Why not userIdHash-onchain immediately
Moving all balances to `balances[userIdHash]` introduces a larger redesign:
- new authorization model
- new deposit/withdraw semantics
- more operator responsibility
- higher complexity
- more risk during transition

The stable Main Wallet approach solves the product issue with much lower complexity.

## 6. Account and Wallet Model
### 6.1 Account
A FUD account is the primary identity in the product.

Fields:
- `user_id`
- `privy_user_id`
- `main_wallet_address`
- `status`

### 6.2 Main Wallet
The Main Wallet is the stable wallet of the account.

Properties:
- always exists
- created or recovered through Privy
- canonical owner of the vault balance
- canonical signer for product actions
- stable regardless of login method

### 6.3 Linked Wallets
Linked wallets are additional wallets associated with the account.

Examples:
- MetaMask
- WalletConnect
- Coinbase Wallet
- other future wallets

Linked wallets may be used for:
- login
- funding
- withdrawal destination
- future recovery or export flows

Linked wallets do not:
- define the balance
- override the main wallet
- determine the currently visible account balance
- become the canonical trading wallet in the base model

## 7. Login and Bootstrap Rules
### 7.1 Universal rule
Regardless of how the user signs in:
- Google
- Email
- external wallet

the account must end up with the same structure:
- one stable Main Wallet
- optional linked wallets
- one account-owned balance

### 7.2 Google / Email login
Flow:
1. user authenticates with Privy
2. Privy creates or recovers embedded wallet
3. embedded wallet becomes `main_wallet_address`
4. account is created or restored
5. vault balance is read from `main_wallet_address`

### 7.3 External wallet login
Flow:
1. user authenticates with external wallet through Privy
2. Privy creates or recovers the embedded Main Wallet
3. external wallet is stored as linked wallet only
4. embedded Main Wallet remains canonical
5. vault balance is read from Main Wallet only

Important:
External wallet login must not convert the external wallet into the primary balance owner.

## 8. Source of Truth
### 8.1 Money
Source of truth for user funds:
- `FUDVault` onchain balance of `main_wallet_address`

### 8.2 Product/UI/cache
DB is used for:
- indexing
- cache
- analytics
- faster UI reads
- order history
- market history
- leaderboard
- reserve tracking if needed
- reconciliation tooling

DB is not the source of truth for actual user funds.

## 9. Contract Direction
### 9.1 Keep address-based balances
Maintain address-based vault balances in the short-to-medium term.

This keeps compatibility with:
- current contract model
- current USDC vault pattern
- existing operator execution approach

### 9.2 Required contract evolution
The vault should evolve toward operator/relayer-friendly methods.

Recommended additions:

#### depositFor
```solidity
function depositFor(address account, uint256 amount) external;
```

Purpose:
- allow deposits to credit the Main Wallet address directly
- reduce dependency on the user wallet paying gas
- support relayer/operator flows

#### withdrawBySig
```solidity
function withdrawBySig(
    address account,
    address to,
    uint256 amount,
    uint256 nonce,
    uint256 deadline,
    bytes calldata signature
) external;
```

Purpose:
- user signs with Main Wallet
- operator/relayer submits
- operator pays gas
- funds can be sent to any chosen destination wallet

#### nonces / replay protection
The contract must support nonce-based replay protection for all signed execution flows.

## 10. Gas Model
### 10.1 Product principle
The user should sign, but not be required to hold ETH for core product actions.

Target UX:
- user signs order / withdrawal authorization
- operator or relayer submits transaction
- relayer/operator pays gas

### 10.2 Role split
Privy provides:
- auth
- embedded wallet
- signing
- account identity linkage

FUD infrastructure provides:
- operator
- relayer
- gas sponsorship
- transaction submission
- retries
- nonce management
- monitoring

### 10.3 Outcome
This gives:
- gasless UX for users
- consistent account wallet behavior
- strong compatibility with prediction-market order flows

## 11. Deposit Model
### 11.1 Product goal
Users must be able to fund their account from any wallet.

### 11.2 Funding rule
Funding source wallet may be:
- MetaMask
- Coinbase Wallet
- another external wallet
- onramp
- bridge
- any valid source

But credited balance must end up under:
- `main_wallet_address`

### 11.3 Recommended deposit model
Preferred direction:
1. user selects “Add funds”
2. system shows Main Wallet / account funding route
3. system supports funding from any source
4. deposit is ultimately credited to Main Wallet balance in vault

Recommended implementation direction:
Either:
- fund Main Wallet directly and then deposit into vault
or
- use `depositFor(main_wallet_address, amount)` style flow through operator/processor

Important product rule:
Funding wallet and balance owner are different concepts.

Any wallet may fund.
Only Main Wallet owns balance.

## 12. Withdrawal Model
### 12.1 Product goal
Users should be able to withdraw to any chosen wallet without needing gas.

### 12.2 Flow
1. user initiates withdrawal
2. user selects destination wallet
3. backend prepares EIP-712 payload
4. Main Wallet signs authorization
5. operator submits `withdrawBySig`
6. relayer/operator pays gas
7. USDC is sent to destination address

### 12.3 Security requirements
- nonce-based replay protection
- deadline on signed payload
- strict domain separation
- optional risk checks / withdrawal holds / limits

## 13. Bets and Order Signing
### 13.1 Canonical signer
All bets must be signed by the Main Wallet.

Not by:
- whichever browser wallet is connected
- whichever wallet the user used to log in today
- whichever wallet extension is available

### 13.2 Execution model
1. backend builds EIP-712 typed data
2. Main Wallet signs
3. operator receives signed order
4. operator executes matching / market logic
5. settlement occurs through the system design
6. DB reflects indexed order state

This keeps:
- signing tied to stable account identity
- execution offchain where appropriate
- settlement onchain where required

## 14. UI and Product Rules
### 14.1 What the user should see
The user should understand:
- “I have a FUD account”
- “This account has a Main Wallet”
- “This account has a balance”
- “I can fund it from any wallet”
- “Linked wallets do not change my balance”

### 14.2 What the user should not see
Avoid:
- wallet mismatch warnings tied to linked wallets
- balance changing when a browser wallet changes
- connect-wallet prompts just to view balance
- any implication that browser wallet session defines the account balance

### 14.3 Recommended UI language
Main Wallet
Used for balances, signatures, and account operations.

Linked Wallet
Can be used for login, funding, or receiving withdrawals.

Add funds
Fund your account from any wallet.

Withdraw
Choose where to receive funds.

## 15. Remove Wallet / Main Wallet UX
If the product later exposes wallet removal, this must be a serious account action, not a simple disconnect button.

For Main Wallet:
Use wording like:
- Remove Main Wallet from account

Not:
- Disconnect wallet

Before allowing removal, require:
- no unsafe balance state
- no unresolved recovery state
- strong warning
- export/recovery acknowledgment if relevant

Example warning:
Removing your Main Wallet may permanently affect access to your account if you have not exported or secured it.

## 17. API Direction
POST /api/users/bootstrap
Purpose:
- create or restore account
- ensure stable Main Wallet
- link optional external wallet metadata

Request example:
```json
{
  "privy_user_id": "did:privy:123",
  "auth_method": "google",
  "email": "user@example.com",
  "embedded_wallet_address": "0xMAIN...",
  "external_wallet_address": "0xMETA..."
}
```

Behavior:
- if user does not exist, create it
- set `main_wallet_address = embedded_wallet_address`
- mark embedded wallet as main
- if external wallet exists, store as linked wallet
- never replace main wallet because external wallet was used for login

GET /api/users/me/wallet
Purpose:
- return Main Wallet info
- return cached balance
- return linked wallets

POST /api/withdrawals/request
Purpose:
- create signed withdrawal request flow

GET /api/users/me/balance
Purpose:
- return account balance derived from Main Wallet

## 18. Migration Strategy
Phase 1
Product migration:
- stop using browser wallet as balance owner
- introduce `main_wallet_address`
- ensure all reads come from Main Wallet
- treat external wallets as linked only

Phase 2
Execution migration:
- add relayer-driven withdrawal flow
- Main Wallet signs, operator submits

Phase 3
Deposit optimization:
- add `depositFor`
- improve friction around direct deposit handling

Phase 4
Balance sophistication:
- split available vs reserved
- improve reconciliation
- optimize settlement batching

## 19. Tradeoffs
Benefits
- solves wallet-switch balance problem
- preserves onchain source of truth for funds
- remains compatible with Privy
- supports gasless user UX
- supports funding from any wallet
- close to Polymarket-style account wallet model
- avoids full custodial DB-ledger design

Costs
- requires disciplined Main Wallet model
- requires relayer/operator infrastructure
- requires contract evolution for best UX
- requires careful deposit/withdraw flow design
- DB and onchain sync still need robust monitoring

## 20. Rejected Alternatives
A. Balance by userIdHash onchain
Rejected for now because:
- larger rewrite
- more complex permissions
- more operator authority
- more migration risk

B. Balance in DB as source of truth
Rejected as primary model because:
- custodial de facto
- weaker crypto-native guarantees
- less independently auditable
- worse fit for desired architecture

C. Browser wallet as current balance owner
Rejected because:
- breaks account consistency
- creates fatal wallet-switch bugs
- confuses users
- makes balance session-dependent

## 21. Final Recommendation
FUD should adopt this architecture:

One account = one stable Main Wallet created/recovered through Privy.
That Main Wallet is the canonical owner of the onchain vault balance.
External wallets are linked tools for login, funding, and withdrawals, but they do not define the balance.
The backend indexes and enhances UX, while operator/relayer infrastructure pays gas and submits transactions.

This gives FUD:
- account-based balance at the product level
- onchain balance at the contract level
- gasless UX
- wallet-independent funding
- strong compatibility with Privy
- a path that stays crypto-native without a full contract rewrite

## 22. One-line version
Do not move the real balance to DB. Do not let the browser wallet define the balance. Make the Main Wallet the stable owner of funds for the account.
